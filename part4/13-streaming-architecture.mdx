Claude Code's interaction model is built entirely on streaming. This chapter dissects the architecture that makes real-time, low-latency Agent interaction possible: from `QueryEngine`'s lifecycle management, to `StreamingToolExecutor`'s concurrency control, to the startup optimization journey that cut cold-start time by 59%.

## QueryEngine: query lifecycle management

### The case for a class

`QueryEngine` owns the query lifecycle and session state. One session corresponds to one instance. Each time a user sends a message, `submitMessage()` opens a new turn while state — messages, file cache, usage statistics, permission denial records, and discovered skill names — persists across turns.

**Why a class instead of functional APIs?** State management complexity. If session state is threaded through function parameters, every new state field requires modifying all function signatures. A class encapsulates state as instance properties: adding a field means updating the constructor, not every call site. It also binds lifetime to the instance — destroying the instance destroys the state, avoiding the memory-leak risks of global state.

`QueryEngine` embodies the **single ownership** principle: one owner for session state. In concurrent scenarios this matters — if multiple components modify the message list simultaneously, messages can arrive out of order or be processed twice. The class boundary provides a natural mutual-exclusion surface.

<Note>
`QueryEngine`'s core query loop is analyzed in detail in Chapter 2 "The Dialog Loop — Heartbeat of an Agent". This chapter focuses on its streaming characteristics and performance optimization mechanisms.
</Note>

### Token stream reception

`submitMessage` is an `AsyncGenerator`. Callers consume streaming messages in real time without waiting for the entire query to complete.

Inside the query loop, streaming events are processed one by one based on their type:

| Event | Action |
|-------|--------|
| `message_start` | Reset the current message's usage counter |
| `message_delta` | Update usage and capture `stop_reason` |
| `message_stop` | Accumulate the current message's usage into the session total |

This phased design enables the system to process tool calls and progress messages in real time without waiting for complete responses.

**The value of streaming in numbers.** Suppose the model generates a response with three tool calls, total output time 5 seconds:

| Strategy | t=1s | t=3s | t=5s | Total |
|----------|------|------|------|-------|
| Non-streaming | Waiting | Waiting | Execute all three tools | 5s + tool execution |
| Streaming | Execute tool 1 | Execute tool 2 | Execute tool 3 | 5s (parallel) |

In streaming mode, tool 1 starts at second 1. By second 5, all three tools may have finished.

### Immediate detection of tool call blocks

When the API returns a `tool_use` content block, the system doesn't wait for the full response before starting to work. Using `content_block_start` and `content_block_delta` events, the system detects an incoming tool call immediately and can pre-look up tool definitions and prepare permission-check context.

Subsequent `content_block_delta` events build the tool's JSON input parameters through an **incremental JSON parser** — parsing while receiving. Since tool `inputSchema` is known in advance, the parser can make reasonable predictions about incomplete JSON and begin pre-warming (for example, triggering OS-level file cache prefetching while the path parameter is still streaming in).

<Warning>
Don't perform heavy computation on delta events during streaming. Each delta may contain only one or two tokens. If parsing logic itself costs more time than it saves, it's counterproductive. The correct approach: lightweight buffer accumulation, with substantive processing triggered only at key boundary events like `content_block_stop`.
</Warning>

### Complete streaming data flow

The full path from user input to output:

```
User input
    │
    ▼
submitMessage() — create new turn
    │
    ▼
Build API request parameters
    │
    ▼
Send Streaming Request
    │
    ▼
Event-by-event processing loop
    ├── message_start ──────────► Reset usage counter
    ├── message_delta ──────────► Update accumulated tokens
    └── content_block_*
            ├── text_block delta ──► yield text stream to UI layer
            └── tool_use_block ────► Prepare StreamingToolExecutor
                                          │
                                          ▼ content_block_stop
                                     Parallel tool execution
                                          │
                                          ▼
                                    Collect tool results
                                          │
                               ┌──────────┴──────────┐
                               │                     │
                          end_turn              tool_use continues
                               │                     │
                          yield complete        Append tool_result,
                                               next loop iteration
```

The key insight: **streaming is not an optimization, it is an architectural choice.** It mandates that every component be designed according to "incremental, interruptible, composable" principles. Streaming is the cornerstone of the entire Agent Harness.

## StreamingToolExecutor: concurrency control

### Execute on arrival

`StreamingToolExecutor`'s design goal: start executing tools immediately when streaming indicates they are ready, rather than waiting for all tool call blocks before processing.

Think of it as a professional kitchen: the chef (model) calls out instructions — "chop onions," "boil water," "make sauce." The traditional approach waits until all instructions are given before starting; "execute on arrival" means each task starts as soon as it's called. Chopping onions and boiling water can happen simultaneously (concurrency-safe), but "add onions to pot" must wait until "onions are chopped" (serial dependency).

Each tracked tool moves through a four-state machine:

<Tabs>
  <Tab title="State machine">

| State | Description | Transition |
|-------|-------------|------------|
| `queued` | Added to executor, waiting | → `executing` when `canExecute = true` |
| `executing` | Running | → `completed` on success; → `cancelled` on Bash failure |
| `completed` | Done, awaiting output | → `yielded` when order constraint satisfied |
| `yielded` | Result delivered | Terminal |

  </Tab>
  <Tab title="Cancellation">

Only Bash tool errors trigger cascading cancellation. When a Bash tool fails, `StreamingToolExecutor` marks it as errored and aborts all parallel sibling tools through `siblingAbortController`.

Bash commands often have implicit dependency chains — if `mkdir` fails, subsequent commands that use that directory are meaningless. Failures of independent tools like `Read` or `WebFetch` don't affect other parallel tools.

  </Tab>
</Tabs>

### Concurrency safety: parallel vs serial

The core logic of concurrency control lives in the `canExecuteTool` method:

- **When no tools are executing** — any tool can start.
- **When tools are executing** — parallel execution is only allowed when the new tool and all executing tools are concurrency-safe.
- **Non-concurrency-safe tools** must execute exclusively; no other tool can run in parallel.

<CardGroup cols={2}>
  <Card title="Concurrency-safe (parallel)" icon="bolt">
    **Read, Grep, Glob, WebFetch**

    Pure read operations with no side effects. These tools can execute simultaneously in any combination.
  </Card>
  <Card title="Non-concurrency-safe (serial)" icon="lock">
    **Bash, Edit, Write, NotebookEdit**

    Tools that modify filesystem or file contents. These must execute in a serial queue, one at a time.
  </Card>
</CardGroup>

**Why not finer-grained dependency analysis?** This is an intentional simplification. Theoretically you could analyze data dependencies between two Bash commands. But that requires understanding shell semantics — costly and error-prone. Claude Code chose a conservative but reliable strategy: Bash is inherently unsafe. Better to wait an extra second than risk parallel execution of potentially conflicting commands.

<Tip>
When designing custom tools, set `isConcurrencySafe` correctly. A common mistake is marking a database-read-only tool as unsafe, causing unnecessary serial waiting. Marking a state-modifying tool as safe is worse — it will cause data races. **When uncertain, choose `isConcurrencySafe: false` (fail-closed).**
</Tip>

### Result buffering and ordered output

Although tools can execute in parallel, results must be output in order. The `getCompletedResults()` method guarantees this through the `yielded` state marker: completed results are yielded sequentially in the order tools were added. If an earlier tool is still executing, subsequent tool results wait even if they have completed.

**Why strict ordering?** The model's reasoning depends on result order. If the model called Read on file A then file B, and B's result arrives first, the model might mistake B's content for A's. Order guarantee keeps the model's view of the world consistent.

Progress messages are an exception — they are immediately yielded without order constraints, so users see real-time feedback from tool execution.

This is like an airport baggage carousel: luggage (tool results) may be processed at different speeds, but passengers (the model) see them in the same order as check-in.

## Startup performance optimization

Claude Code's startup time directly affects user experience. CLI psychology research shows startup delays over 2 seconds make users feel the tool is "clunky," and over 5 seconds triggers "is it frozen?" anxiety.

Three optimization strategies work together as a system:

**Identify critical path → Parallelize non-critical paths → Defer non-essential modules → Eliminate dead code**

### Parallel prefetching: racing against module evaluation

At the top of Claude Code's entry file, the system starts two subprocesses at the earliest stage of module evaluation — letting them execute in parallel with the subsequent ~65ms import chain.

**`startMdmRawRead`** spawns multiple `plutil` processes in parallel on macOS to read MDM plist configurations (or `reg query` on Windows). It uses `existsSync` to quickly skip non-existent plist files, avoiding unnecessary 5ms spawn overhead.

**`startKeychainPrefetch`** changes two keychain reads (OAuth tokens ~32ms, legacy API key ~33ms) from serial to parallel, executing in parallel with the import chain.

Both prefetch results are awaited in subsequent initialization stages — at which point the subprocesses are likely already complete, with almost zero wait.

**The numbers:**

| Phase | Serial (before) | Parallel (after) |
|-------|-----------------|------------------|
| Import chain | 65ms | 65ms |
| MDM read | +30ms | 0ms (parallel) |
| Keychain #1 | +32ms | 0ms (parallel) |
| Keychain #2 | +33ms | 0ms (parallel) |
| **Total** | **160ms** | **65ms** |

**Result: 160ms → 65ms, saving 95ms (59% improvement).**

95 milliseconds may not sound like much, but in the CLI world users are extremely sensitive to the gap between "typing command" and "seeing output." This strategy essentially utilizes idle CPU time while waiting for I/O.

<Note>
Conditions for parallel prefetching to be worthwhile: (1) the operation is I/O-intensive, not CPU-intensive; (2) the result won't be used immediately but is needed at a later point; (3) prefetch failure won't block the main flow. Violate any condition and parallelization adds complexity without benefit.
</Note>

### Lazy loading strategies

<AccordionGroup>
  <Accordion title="Load on first use (heavy dependencies)">
    The `messageSelector` in `QueryEngine` uses `require()` for true lazy loading because the `MessageSelector` component pulls in heavy dependencies like React/Ink. These are loaded only when first needed, not at module initialization time.

    The `lazySchema` pattern extends this to tool input and output schemas: Zod schema construction is deferred until first actual use. A moderately complex schema with nested objects and descriptions takes 0.1–0.3ms to construct. Claude Code has 60+ tools — building all schemas at startup would add 6–18ms of unacceptable overhead.

    `lazySchema` leverages JavaScript's getter semantics: first access triggers construction and caches the result as an instance property. Subsequent accesses read directly from cache. This is a "trigger once, permanently replace" pattern.
  </Accordion>
  <Accordion title="Conditional imports (feature-gated modules)">
    Coordinator mode and snip compaction achieve dead-code elimination through feature flag gating functions. When feature flags are off, bundlers completely eliminate these code paths and their entire dependency trees, reducing bundle size and module initialization time.
  </Accordion>
  <Accordion title="Compile-time elimination">
    Bun's `bun:bundle feature()` wrapping allows code blocks to be entirely absent from build output when disabled — no branch prediction penalty, no runtime check overhead. For disabled features, the code simply doesn't exist in the binary.
  </Accordion>
</AccordionGroup>

**Decision rule for lazy loading:** If a module's load time exceeds 5ms and it isn't needed in the first 3 seconds after startup, it should be lazy-loaded. Below 5ms, the runtime check overhead may offset the benefit.

### Startup optimization checklist

Use this checklist when building your own CLI Agent tool:

- [ ] **Identify critical path** — use `console.time` or profiling tools to measure each initialization step
- [ ] **Parallelize I/O operations** — file reads, network requests, and process spawns should execute in parallel with module loading
- [ ] **Lazy load heavy dependencies** — React, large parsers, database drivers should be deferred to first use
- [ ] **Eliminate unused code** — use feature flags and tree-shaking to remove unneeded features
- [ ] **Lazy initialize heavy objects** — Zod schemas, regexes, and WASM modules should defer construction
- [ ] **Measure and iterate** — establish CI baselines for startup time to prevent performance regression
- [ ] **Focus on P95, not average** — startup experience on slow machines is the real user experience

## Token cost tracking

### The two-layer cost engine

Claude Code's cost tracking is built on two core functions: `updateUsage` and `accumulateUsage`.

**`updateUsage`** handles usage data from single streaming events. Since Anthropic API's `message_delta` events only return incremental fields, `updateUsage` uses `> 0` guards for each field to distinguish "not returned" from "truly zero."

This guard is critical: if the API returns `input_tokens: 0` because the request was entirely cache-served, the system won't incorrectly overwrite the previously recorded true value to zero.

The essence of the `> 0` guard is a **monotonic increase guarantee**: usage fields only increase, never decrease or get overwritten with smaller values.

<Warning>
Don't use `|| 0` as a default value in `updateUsage`. The `||` operator treats `0`, `undefined`, `null`, and `""` all as falsy, causing true zero values to be incorrectly replaced. Use a `> 0` guard (or `!== undefined && val > 0`) to correctly distinguish "not returned" from "truly zero."
</Warning>

**`accumulateUsage`** accumulates usage across messages through simple field summation — input tokens, cache creation tokens, cache read tokens, output tokens, etc.

In `QueryEngine`, accumulation happens at each `message_stop` event. In forked agents, accumulation happens at each `message_delta` event. The difference: `QueryEngine` accumulates per message (sufficient for a long-lived session manager), while forked agents accumulate per event. If a fork crashes before `message_stop`, per-message accumulation would lose the entire message's usage data.

### Budget awareness strategy

<AccordionGroup>
  <Accordion title="Token budget thresholds">
    Set a token budget at the start of each session. Claude Code implements this through the `maxBudgetUsd` parameter.

    | Budget level | Suggested action |
    |-------------|-----------------|
    | < 50% | Normal operation, no alert |
    | 50–80% | Light reminder (status bar) |
    | 80–95% | Clear warning (suggest compacting context) |
    | > 95% | Strong warning (limit new tool calls) |
    | ≥ 100% | Hard limit (stop dialog loop) |
  </Accordion>
  <Accordion title="Layered cost tracking dimensions">
    Don't just track totals. Break costs down to locate hotspots:

    - **By turn** — which turn consumed the most tokens? Usually the initial context-building turn.
    - **By tool** — which tool's execution result is largest? Large file reads are the most common token black holes.
    - **By type** — input vs output token ratio? Output tokens are usually 3–5× more expensive.
    - **Cache efficiency** — what is the cache hit rate? Below 70% usually indicates context management problems.
  </Accordion>
  <Accordion title="Cost anomaly signals">
    Set normal ranges and alert on deviation:

    - Single turn input exceeds 100K tokens — a tool probably returned an oversized result
    - Single turn output exceeds 10K tokens — the model may be producing verbose output
    - Cache hit rate drops suddenly from 80% to 20% — context may have been unexpectedly cleared

    **Key insight:** Tool output token costs are often more worth watching than model output. A large file read might produce 60K+ cache creation tokens, while a model turn output is usually only 1–5K tokens.
  </Accordion>
</AccordionGroup>

## Caching optimization strategies

### Five dimensions of the cache key

Anthropic API's prompt caching reduces costs and latency significantly. The cache key consists of five dimensions:

<CardGroup cols={2}>
  <Card title="System prompt" icon="terminal">
    Includes the system prompt, user context, and system context. Any change here invalidates the entire cache prefix.
  </Card>
  <Card title="Tools" icon="wrench">
    The tool use context. Adding, removing, or reordering tools breaks the cache.
  </Card>
  <Card title="Model" icon="cpu">
    The model identifier. Switching models creates a new cache entry.
  </Card>
  <Card title="Messages prefix" icon="messages">
    The fork context messages — shared prefix between parent and fork requests for cache reuse.
  </Card>
</CardGroup>

The fifth dimension — **thinking config** (`budget_tokens`, thinking toggle) — has a subtle trap: if a fork sets `maxOutputTokens`, related logic clamps `budget_tokens`, causing thinking config changes and thus cache invalidation. Source comments explicitly warn: if a fork uses `cacheSafeParams` to share the parent request's prompt cache, different `budget_tokens` will invalidate the cache because thinking config is part of the cache key.

**Economic impact of cache hits** (Claude 3.5 Sonnet pricing):

| Token type | Standard price (per 1M) | Cache hit price |
|------------|------------------------|-----------------|
| Input token | $3.00 | — |
| Cache write token | $3.75 (+25%) | — |
| Cache read token | — | $0.30 (−90%) |
| Output token | $15.00 | $15.00 (unchanged) |

A typical 10-turn session sending 200K input tokens: without caching costs $0.60; with 80% cache hit rate costs $0.168 — a **72% reduction**. At scale, this can mean hundreds of dollars per day.

### Byte-level consistency in fork mode

`runForkedAgent` concatenates `forkContextMessages` with `promptMessages` as the initial message list when starting a fork. This ensures the fork request shares exactly the same message prefix as the parent request, hitting the parent's prompt cache.

`contentReplacementState` cloning further guarantees wire-format consistency: the fork makes the same replacement decisions for `tool_use_ids` in parent messages, producing the same byte-level prefix.

**Byte-level consistency is stricter than logical equivalence.** Two JSON objects can be logically equal (same key-value pairs) but byte-different (different key order, different whitespace). The API's cache key is based on exact byte prefix matching — even a tiny format difference invalidates the cache.

This is like two documents with identical content but different formatting: same information, different hash. Cloning `contentReplacementState` is not over-engineering; it's an inevitable requirement of the caching mechanism.

### Practical cache optimization strategies

<AccordionGroup>
  <Accordion title="Strategy 1: minimize system prompt changes">
    System prompt is the beginning of the cache prefix — any change invalidates the entire cache. Common culprits:

    - **Timestamp injection** (different every request) — anti-pattern; use relative time
    - **Dynamic tool list order** — sort before injecting to guarantee stability
    - **User preference format differences** — standardize the format
  </Accordion>
  <Accordion title="Strategy 2: message order stability">
    Message addition order directly affects cache prefix. If a "system notification" message is inserted between dialog messages, the prefix breaks. Claude Code avoids this by placing attachment messages (`AttachmentMessage`) in specific positions — system notifications never interleave with dialog messages.
  </Accordion>
  <Accordion title="Strategy 3: cache preheating">
    Proactively include content in the first request that subsequent requests may share. If you know a fork agent will be triggered after the turn (like a post-turn summary), consider its `cacheSafeParams` needs when building the parent request so the parent's prefix can be reused.
  </Accordion>
  <Accordion title="Strategy 4: monitor cache health">
    Recommended baseline metrics for Agent systems:

    | Metric | Healthy range | Anomaly signal |
    |--------|--------------|----------------|
    | Overall cache hit rate | > 70% | System prompt changing frequently |
    | Fork cache hit rate | > 85% | Cache-safe params inconsistent |
    | Cache create/read ratio | < 0.3 | Large number of new prefixes being created |
    | Single request cache creation tokens | < 50K | Context too large, insufficient compression |
  </Accordion>
</AccordionGroup>

<Note>
Context compression strategy directly affects cache efficiency. Each compression rewrites message history, potentially causing cache breaks. Designing compression strategies requires balancing "releasing token space" with "maintaining cache continuity."
</Note>

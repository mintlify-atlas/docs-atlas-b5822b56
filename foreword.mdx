## Yù Yú — from building chariots to building agents

> *"Of all instruments, the chariot demands the most hands to build."*
> — *Kǎo Gōng Jì* (Rites of Zhou, c. 300 BC)

Two thousand three hundred years ago, the authors of the *Kǎo Gōng Jì* wrote that sentence. In the pre-Qin era, the horse-drawn chariot was the most complex engineered system humanity had ever built — bar none.

How many trades did it take to build a chariot? Woodworkers constructed the **yú** (舆, the carriage body); metalworkers cast the **wèi xiá** (軎辖, the axle pins); leather workers fashioned the **pèi** (鞁, the harness); lacquerers finished the surfaces; and wheel-wrights shaped the **fú** (辐, the wheel spokes). The *Kǎo Gōng Jì* declared that "heaven provides the seasons, earth provides its energy, materials provide their virtue, craftsmen provide their skill" — only when all four converge does a fine chariot emerge.

Each component has deep significance, and they map across two millennia onto the Agent Harness architecture dissected in this book:

| Ancient chariot | Meaning | Agent Harness |
|---|---|---|
| **舆** (yú) | The carriage body — the core structure that carries the passenger | **Harness runtime** — the engineering framework that carries the LLM |
| **辕** (yuán) | The shafts — set direction, transmit force | **Dialog loop** — the main loop that drives the agent forward |
| **辐** (fú) | Wheel spokes — connect the axle hub to the outer rim | **Tool system** — the bridge connecting the LLM to the external world |
| **軎辖** (wèi xiá) | Axle pins — hold the wheels on, prevent them falling off | **Permission pipeline** — the safety mechanism that constrains agent behavior |
| **轼** (shì) | The front crossbar — passengers hold it as a gesture of respect | **Hook system** — ceremonial extension points in the lifecycle |
| **御** (yù) | The coachman's art of driving | **Architectural knowledge** — the ability to understand and master Agent systems |

Confucius used the chariot to illustrate trustworthiness: *"A large cart without its linchpin, a small cart without its axle-pin — how can it move?"* A cart missing its retaining pin cannot travel, just as an agent without permission guardrails cannot be trusted. The *Kǎo Gōng Jì* recorded that the carriage shaft must be bent to precisely the right degree — too deep and it breaks, too shallow and it cannot bear the load — just as the autonomy of an agent must find the balance between capability and safety.

The oracle-bone character for **舆** (yú) was described by scholar Luo Zhenyu as "showing many hands building a chariot together." Today, building a production-grade Agent system likewise requires many hands: the tool system, the permission pipeline, context management, state persistence, streaming communication, error recovery — every subsystem is one craftsman's contribution, and only together do they let the agent truly get on the road.

**Ancient people drove chariots — mastering the most precise machine on earth. Modern people drive chariots — mastering the most complex intelligent systems of the silicon age.**

This is why the book is named **Yù Yú** (御舆), and why readers call it the "chariot book."

## Why this book was written

### The three waves of AI-assisted programming

Looking back over the past several years, AI-assisted programming has gone through three distinct waves, each profoundly changing the relationship between developers and code.

<Steps>
  <Step title="First wave (2021–2022): the code completion era">
    The launch of GitHub Copilot marked the formal entry of AI into developers' daily workflows. The core paradigm was **inline completion** — AI predicted the next line or code block based on the current file context, and developers accepted suggestions with the Tab key.

    This was a highly passive, highly localized mode of assistance: AI saw dozens of lines of code around the cursor and produced fragment-level suggestions. It could not reason across files, understand project structure, or proactively execute operations.
  </Step>
  <Step title="Second wave (2023–2024): the conversational assistant era">
    With expanding context windows and multi-file awareness, AI tools evolved from "completion boxes" to "dialog boxes." Editor-embedded tools like Cursor, Windsurf, and Continue flourished. Developers could describe requirements in natural language, and AI would generate code across multiple files.

    But AI at this stage was still constrained by the editor boundary — it could write code but not run it; suggest tests but not execute them; identify problems but not verify fixes. Developers constantly switched between AI and the terminal, acting as human glue.
  </Step>
  <Step title="Third wave (2025 to present): the autonomous agent era">
    The paradigm shift we are experiencing now is far more profound than the previous two. AI is no longer "an assistant sitting in the editor waiting for your questions," but "an agent that autonomously executes tasks in the terminal."

    It can directly run shell commands, read and write the file system, execute test suites, operate Git version control — and when encountering errors, autonomously adjust strategy and iteratively fix them. From simple code completion to multi-file refactoring, from single-turn Q&A to cross-tool orchestration, AI programming assistants are undergoing a fundamental paradigm shift from chatbot to Agent.
  </Step>
</Steps>

### Agent Harness: a new architectural concept

During this transition, a key architectural pattern has surfaced: **Agent Harness** — a runtime framework built around an LLM, responsible for managing tool registration and dispatch, permission control, state persistence, streaming output, error recovery, and other cross-cutting concerns.

The LLM is an immensely powerful force that knows nothing of the outside world on its own. Agent Harness is the chariot built for this force: the tool system is the **spokes** (辐) that transmit energy to the ground; the permission pipeline is the **axle pin** (軎辖) that ensures the wheels stay on track; context compression is the elasticity of the **shaft** (辕) — bearing maximum load in a finite space; streaming communication is the turning of the wheels, keeping everything in motion.

And you — the developer who has understood this architecture — are the **coachman** (御者). The master coachman does not breed the horse, nor build the cart, but understands the structure of the yú and the tension of the reins, and so drives with complete control.

<Info>
  Agent Harness is not an SDK, not an API wrapper, and certainly not simple prompt engineering. It is the engineering infrastructure that enables an LLM to truly "do things." If the LLM is the agent's brain, then the Agent Harness is the agent's skeleton, muscles, nerves, and immune system.
</Info>

### The March 31, 2026 disclosure that started a conversation

In early 2025, Anthropic released Claude Code — an AI programming agent that runs directly in the terminal without depending on a specific editor or graphical interface. This choice was itself a declaration: an Agent does not need the constraints of a GUI; it needs a fully-featured runtime environment.

<Info>
  On March 31, 2026, security researcher [Chaofan Shou (@Fried_rice)](https://x.com/Fried_rice) discovered that the `@anthropic-ai/claude-code` package published on the npm registry contained a source map file referencing a complete source archive in an Anthropic R2 storage bucket — with no access controls. The disclosure tweet received over **17 million views**. The tech community engaged in an unprecedented discussion about Agent architecture. Anthropic subsequently patched the configuration issue.
</Info>

That discussion made it clear: Agent Harness had evolved from an obscure engineering concept into a topic the entire developer community cared about. But the discussions were scattered and fragmented — some focused on tool call design, others on permission models, others on streaming architecture — and no one had assembled the pieces into a complete picture.

This book attempts to fill that gap. It does not rely on any unauthorized materials. Based entirely on Claude Code's public documentation, product behavior, and community discussions, it systematically deduces and explains the design principles of Agent Harness.

## Three characteristics of this book

This book has three core characteristics that distinguish it from other AI-related books.

**Architecture analysis, not API documentation.** This book does not teach you "how to call the Claude API to build a chatbot." Starting from Claude Code's product behavior and public information, it progressively reconstructs the core architecture — how the dialog loop drives execution, how permissions are checked in layers, how context is compressed. Through systematic architectural deduction, it presents the complete design picture of Agent Harness.

**Design philosophy, not usage tutorials.** This book does not list "10 tips to make Claude Code work better." Instead, it discusses the five major design principles of Agent Harness — async-streaming-first, embedded security boundaries, cache-aware design, progressive capability extension, and immutable state flow — and how these principles map to concrete code structures.

**Transferable architectural knowledge.** Each chapter distills universal design patterns that go beyond Claude Code itself. After reading this book, you will not only understand Claude Code's internal mechanisms, but be able to apply this knowledge to your own Agent projects, whether you use LangChain, AutoGen, CrewAI, or build from scratch.
